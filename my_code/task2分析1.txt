一、项目概述

1.1 目标

利用无人机搭载 Orbbec Gemini 336 深度相机，在贴近幕墙垂直上升的过程中，实时检测玻璃面板的框架结构（龙骨/横梁/胶缝），计算每块玻璃面板的实际高度，并自动归类出大厦共有几种尺寸。


1.2 精度要求

测量高度与实际高度误差 ≤ 10%


1.3 作业条件

条件	说明
距墙距离	0.2 ~ 0.5m，实时变化
帧率	10fps（可调）
运动方式	绳索牵引，直上直下，手动控制上升
计算方式	机载实时处理
幕墙类型	明框 / 隐框 / 半隐框均可能，同一栋大厦可能混合存在
墙面高度	几十米到几百米
玻璃尺寸	多种，需自动识别归类
扫描范围	先假设只扫一列

1.4 传感器

整套系统仅使用 Orbbec Gemini 336 一台设备，不依赖任何外部传感器。


传感器	提供什么数据	作用
Orbbec 彩色摄像头	RGB 图像帧（MJPG 格式，1280x720）	框架线检测（主力）
Orbbec 深度摄像头	深度图（与彩色 D2C 对齐）	生成 3D 点云
Orbbec 加速度计	三轴加速度 (ax, ay, az)	计算俯仰/横滚角，修正点云
Orbbec 陀螺仪	三轴角速度	辅助（可选）

关键发现（实测数据）：


text
text
彩色帧: 1280x720  格式=5(MJPG)  数据大小=135KB（压缩后）
深度帧: 1280x720  格式=8         数据大小=1.8MB  valueScale=1（单位mm）

IMU 静止朝墙时: ax≈0  ay≈-9.8  az≈0
→ 坐标系: X=右, Y=下, Z=前（朝墙）
→ ay=-9.8 说明 Y 轴与重力方向一致

点云单位: mm（乘以 0.001 转为米）


二、核心原理

2.1 为什么测框架而不测玻璃

text
text
玻璃：透明、有镜面反射、有倒影
      → 边缘检测不可靠
      → 深度传感器会穿透或被打乱

框架（龙骨）：不透光的金属结构
      → 灰度均匀，与玻璃有强烈对比
      → 深度图对金属表面测量准确
      → 几何上是直线，容易检测

核心策略：测量龙骨框架，而不是玻璃本身。框架之间的距离就是玻璃面板的高度。


2.2 三种框架类型

幕墙框架在三种类型下外观不同，但在 0.2~0.5m 的近距离下均可检测：


text
text
明框（金属压条在外）：
  ██████████████████████████  ← 金属压条，宽 50~150mm
  ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  ← 深色，灰度值低且均匀
  ██████████████████████████
  特征：宽暗带，边缘锐利
  检测难度：低

隐框（框架在玻璃背面）：
  ···························  ← 玻璃表面（高亮，有反射）
  ─────────────────────────── ← 胶缝，宽 10~20mm，深色
  ···························
  特征：窄暗线，宽度一致，贯穿画面
  检测难度：中等（近距离下可行）

半隐框（一个方向明框，一个方向隐框）：
  ═══════════════════════════  ← 横向明框（宽暗带）
  ┃                         ┃ ← 竖向隐框（窄暗线）
  ═══════════════════════════
  检测难度：看方向

2.3 为什么用 3D 点云

之前的方案需要多个外部传感器（超声波、绳索编码器、独立 IMU）来做标定。Orbbec 336 本身提供深度图，可以直接生成 3D 点云，每个像素都有真实的三维坐标 (X, Y, Z)。


text
text
之前（像素坐标 + 外部传感器）：
  摄像头 → 像素坐标
  超声波 → 距离
  绳索编码器 → 高度
  IMU → 修正
  像素 × (像素尺寸 × 距离 / 焦距) = 真实尺寸
  → 需要 5 个传感器，累积误差大

现在（3D 点云）：
  Orbbec → 深度图 → 点云（每个像素有真实 3D 坐标）
  → Y 坐标就是墙面上的真实高度
  → 不需要焦距、像素尺寸、距离换算
  → 只需要 Orbbec 一台设备 + 内置 IMU

2.4 整体测量过程

text
text
无人机从底部升到顶部，每一帧问一个问题：

  "画面里有没有一条水平的暗线？"

  有 → 把它在点云中的 3D Y 坐标记录下来
  没有 → 继续上升

升完之后，把所有记录到的 Y 坐标排个序
相邻两个的差值，就是玻璃面板的高度


三、框架线检测算法

3.1 检测思路：水平灰度投影法

对图像每行取平均灰度，得到一维信号。框架/胶缝所在行的平均灰度明显低于周围，形成"谷值"。


text
text
明框的灰度投影：                    隐框的灰度投影：

灰度                                灰度
 ▓▓▓▓▓                            ▓▓▓▓▓▓▓▓▓
 ▓▓▓▓▓                            ▓▓▓▓▓▓▓▓▓
 ▓▓▓▓▓___________                 ▓▓▓▓▓▓▓▓▓
 ▓▓▓▓▓           \  ← 宽谷        ▓▓▓▓▓▓▓▓▓\  ← 窄谷
 ▓▓▓▓▓           /    (明框)       ▓▓▓▓▓▓▓▓▓/    (胶缝)
 ▓▓▓▓▓_________/                  ▓▓▓▓▓▓▓▓▓
 ▓▓▓▓▓                            ▓▓▓▓▓▓▓▓▓

两种谷都找得到 ✓                    也能找到 ✓
谷宽度不同 → 可以区分类型

3.2 双重检测方法

text
text
方法1: 水平灰度投影（主）
  · 每行取平均灰度 → 一维信号
  · 去趋势（消除整体亮度渐变）
  · 找暗谷 → 框架/胶缝位置
  · 谷宽度 → 区分明框（宽）vs 隐框（窄）

方法2: 垂直 Sobel + 水平形态学（辅）
  · 垂直 Sobel 检测水平边缘
  · 宽水平核做闭运算，连接水平边缘
  · 在结果上做水平投影，找峰值

两种方法合并，位置相近的候选取最强的


四、3D 点云高度计算

4.1 点云生成

text
text
Orbbec SDK 提供 PointCloudFilter:
  · 输入: 彩色帧 + 深度帧（D2C 对齐后）
  · 输出: 点云数组，每个点有 (x, y, z) 坐标
  · 单位: mm（乘以 0.001 转为米）

点云的坐标系:
  X = 右
  Y = 下
  Z = 前（朝墙方向）

Z 值就是到墙的距离（0.3m 的墙 → Z ≈ 300mm）
Y 值就是相对于摄像头的高度偏移

4.2 IMU 修正

无人机可能有俯仰/横滚，导致点云坐标与重力方向不对齐。需要用 IMU 数据修正。


text
text
IMU 数据来源: Orbbec 内置加速度计
静止朝墙时: ax≈0, ay≈-9.8, az≈0
  → Y 轴 = 重力方向（向下为正）

修正方法:
  · 读取加速度值
  · 低通滤波去噪
  · 计算 pitch = atan2(az, -ay)
  · 计算 roll  = atan2(-ax, -ay)
  · 对点云做旋转，撤销 pitch 和 roll
  · 修正后 Y 方向严格与重力对齐

修正前后的区别：


text
text
修正前（摄像头有俯仰）：           修正后（重力对齐）：

  墙面点的 Y 坐标                墙面点的 Y 坐标
  分布是斜的                     分布是水平的

  Y ↑  ╱                       Y ↑
    │ ╱                           │ ─────────
    │╱                            │ ─────────
    └──→ Z                        └──→ Z

4.3 框架线 3D 高度获取

检测到一条框架线在画面中的像素位置 y_pixel 后，从点云中获取真实高度：


text
text
1. 取框架线所在行（±2 行）的点云数据
2. 沿水平方向每隔若干像素采样
3. 对每个采样点做 IMU 旋转修正
4. 取 Y 坐标的中值（中值比均值更鲁棒，不受个别异常点影响）
5. 得到这条框架线的 3D Y 坐标

4.4 绝对高度计算

单帧的 Y 坐标是相对于摄像头的。需要帧间位移累积来得到绝对高度：


text
text
原理：
  帧 k:   同一面墙在点云中 Y = -0.50m
  帧 k+1: 同一面墙在点云中 Y = -0.47m
  → 摄像头上移了 3cm

实现：
  每帧计算墙面点云的中值 Y（经过 IMU 修正）
  帧间位移 = 当前中值 - 上一帧中值
  累积位移 += 帧间位移（合理范围内）

框架线绝对高度 = 累积位移 - 框架线的 Y 坐标
  （Y 向下为正，高度向上为正，所以取负号）

5.2 去重记录

同一条框架线在无人机上升过程中会被多帧检测到。用绝对高度去重，多次检测取加权平均：


text
text
记录策略：
  · 新检测到一条框架线时，与已记录的所有线比较绝对高度
  · 如果差值 < 4cm → 认为是同一条线，更新（加权平均）
  · 如果差值 ≥ 4cm → 认为是新的框架线，新增记录

效果：
  · 同一条线被检测 30 次 → 精度远高于只检测 1 次
  · 偶发的误检被大量正确检测淹没

5.3 自动归类

text
text
归类策略：
  · 高度差 < 6cm 的面板 → 认为是同一种尺寸
  · 统计每种尺寸出现的次数
  · 自动标注:
      出现 ≥ 3 次 → "标准层"
      高度 > 3.0m → "大堂层"
      高度 < 0.8m → "设备层"
      其他        → "非标准"

六、特殊情况处理

6.1 MJPG 格式解码

text
text
问题：Orbbec Gemini 336 的彩色输出是 MJPG 压缩格式
     （1280x720，压缩后约 135KB，而非原始的 2.7MB）
     直接当作原始 RGB 访问会导致内存越界崩溃

解决：检测格式类型，MJPG 用 cv::imdecode 解码

6.2 反光导致误检

text
text
问题：强烈反光在灰度投影中可能形成假谷值
解决：
  · 高光区域灰度值接近 255，通过亮度阈值过滤
  · 真实框架的谷值跨越画面大部分宽度，反光通常不规则
  · 多帧投票：真正的框架线会被反复检测到，反光不会

6.3 隐框胶缝极细

text
text
问题：2mm 胶缝在远距离几乎不可见
解决：
  · 0.2~0.5m 距离下，2mm 胶缝 ≈ 5~6px，勉强可检测
  · 自适应阈值，对低对比度区域降低检测阈值
  · 如果实在检测不到，可依赖建筑设计图纸的标准分格尺寸做先验约束

6.4 模糊帧

text
text
问题：手动操作可能导致模糊帧
解决：检测帧的模糊程度（拉普拉斯方差），模糊帧直接丢弃不参与计算

6.5 墙面突出物

text
text
问题：有阳台、空调板、装饰线条等突出物
解决：
  · 点云中 Z 值突然变化 → 前方有突出物
  · 该帧标记为异常，不参与框架线高度计算
  · 等距离恢复正常后继续正常处理

6.6 点云尺寸与图像不匹配

text
text
问题：D2C 对齐后点云尺寸可能与彩色图不同
解决：
  · 启动时诊断打印点云点数与图像像素数
  · 处理前做安全检查，防止越界


七、误差分析

7.1 各环节误差来源

误差来源	量级	控制方法
Orbbec 深度精度	±1~2% 距离	在 0.3m 处约 ±3~6mm
IMU 精度	±0.5°	低通滤波抑制噪声
框架线检测精度	±2~5px	灰度投影 + Sobel 双重检测
帧间位移累积	每帧 ±1~3mm	中值过滤消除偶发偏差
MJPG 解码	无损	imdecode 精确解码

7.2 误差控制策略

text
text
1. 每帧独立标定 → 不依赖运动假设
2. 多帧检测同一条框架 → 取加权平均，抑制随机误差
3. IMU 低通滤波 → 消除高频噪声
4. 点云中值采样 → 不受个别异常点影响
5. 模糊帧过滤 → 防止运动模糊带来的检测错误

7.3 最终精度预期

text
text
最佳情况（各传感器配合好，框架清晰）：误差 3~5%
一般情况（有部分隐框，手动飞行不太稳）：误差 5~8%
最差情况（隐框为主，距离波动大）：误差 8~12%

目标 ≤ 10% 在大多数场景下可达
